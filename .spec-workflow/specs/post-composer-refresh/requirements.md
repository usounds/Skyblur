# 要件定義書

## 概要

投稿画面を刷新し、投稿者が「何を書くか」「伏せた内容を誰に見せるか」「投稿後にどう見えるか」を順番に確認しながら、安心して Skyblur 投稿を作成・編集できる体験にする。

現状の投稿画面は、本文、Bluesky プレビュー、補足、返信、公開設定、返信制限、投稿ボタンが単一カラムで並んでおり、機能は揃っている。一方で、Skyblur の中心価値である「伏せる箇所と見せる相手の判断」がフォーム全体の中に埋もれやすく、特にモバイルでは設定量が多いため投稿前の確認と操作完了までの距離が長い。

この刷新では投稿作成の主要フローを、`書く`、`見せる相手`、`投稿前チェック` の 3 ステップに整理する。単にフォームを分割するのではなく、投稿者の不安を「本文と伏せ字」「閲覧できる相手」「Bluesky/Skyblur 上の見え方」の順に解消することを目的とする。

投稿作成・編集は `/console` 直下の管理画面から分離し、専用の production route で扱う。`/console` は投稿済みコンテンツの管理、アカウント状態、作成画面への導線を中心にし、投稿作成・編集画面は入力状態、離脱保護、投稿前チェックに集中できる構造にする。

パスワード保護は古い制約に引きずられた特別扱いではなく、公開範囲の一種として自然に見せる。ただし本 MVP では保存形式をまたぐ公開範囲変更までは扱わず、既存投稿の編集は同じ保存形式のまま本文・補足・基本設定を編集できる範囲に絞る。パスワード投稿は編集画面に入る前に解除用パスワードで復号し、同じパスワード保護形式のまま編集できるようにする。パスワード保護と他の公開範囲を相互に変更する完全な区分変更は次フェーズの対象とする。

## プロダクト方針との整合性

Skyblur の product steering は「伏せることを第一級の体験にする」「投稿者の意図を守る」「UI の理解しやすさ」を重視している。本要件は、Bluesky に公開される伏せ字本文と Skyblur でのみ参照できる本文・補足の違いを投稿者に明確化し、公開範囲の誤選択を減らすことで、この方針に沿う。

また、アクセス制御は既存の deny-by-default 方針を維持する。刷新対象は主に投稿画面の情報設計と操作体験であり、既存の AT Protocol、OAuth、XRPC、Lexicon、制限付き投稿保存の意味を変えない。要件上も、既存の公開範囲や投稿保存フローに回帰を起こさず、E2E は `/console` を起点に専用の投稿作成・編集 route へ遷移する production route 上の実ユーザーフローを対象にする。

## 要件

### 要件 1: 3 ステップで投稿作成体験を整理する

**ユーザーストーリー:** 投稿者として、投稿作成を「書く」「見せる相手」「投稿前チェック」の順に進めたい。そうすることで、一度に多くの設定を判断せず、安心して投稿できる。

#### 受け入れ基準

1. 投稿者が新規投稿を開始したとき、システムは `書く`、`見せる相手`、`投稿前チェック` の 3 ステップを表示すること。
2. 投稿者が `書く` ステップにいるとき、システムは本文、伏せ字指定、補足に集中できる画面を表示すること。
3. 投稿者が `見せる相手` ステップにいるとき、システムは公開範囲、パスワード、リスト、返信・引用の詳細設定を表示すること。
4. 投稿者が `投稿前チェック` ステップにいるとき、システムは Bluesky 上の表示、Skyblur 上の表示、誰が読めるか、補足の扱い、返信・引用設定の要約を表示すること。
5. 投稿者が前のステップに戻る場合、システムは入力済みの本文、補足、公開範囲、詳細設定を保持すること。
6. 投稿者が確認や修正のために任意のステップへ戻る場合、システムは入力済みの内容を失わずに該当ステップを表示すること。
7. 投稿者が入力不備のある状態で次へ進もうとした場合、システムは該当ステップ内で理由を説明し、修正できるようにすること。
8. 投稿者が投稿前チェックで問題を見つけた場合、システムは該当ステップまたは該当項目へ戻れる導線を提供すること。

### 要件 2: 投稿作成・編集ルートを `/console` から分離する

**ユーザーストーリー:** 投稿者として、投稿一覧や管理画面とは分離された専用の作成・編集画面で投稿内容を整えたい。そうすることで、管理と投稿作成の目的が混ざらず、集中して投稿できる。

#### 受け入れ基準

1. 新規投稿作成は `/console` 直下ではなく、専用の production route で開始できること。
2. 既存投稿編集は新規投稿とは区別された専用の production route で開始できること。
3. `/console` は投稿済みコンテンツの管理、アカウント状態、作成画面への導線を中心に扱うこと。
4. 投稿作成・編集画面は `/console` の認証状態、アカウント選択、戻り先、既存導線と一貫して動作すること。
5. 投稿作成・編集画面では、`書く`、`見せる相手`、`投稿前チェック` の 3 ステップを維持すること。
6. モバイルでも `/console` から投稿作成・編集画面へ遷移し、入力途中の操作を継続できること。

### 要件 3: 投稿内容の入力と伏せ字指定を中心に据える

**ユーザーストーリー:** Bluesky にネタバレや限定情報を投稿したいユーザーとして、本文を書きながら伏せる範囲を分かりやすく指定したい。そうすることで、タイムライン上に出したくない情報を誤って公開せずに投稿できる。

#### 受け入れ基準

1. 投稿者が本文を入力しているとき、システムは本文入力、文字数、伏せ字指定操作、伏せ字に関する警告を同じ文脈で確認できるようにすること。
2. 投稿者が本文中の文字列を選択して伏せ字指定を行うとき、システムは選択範囲を `[]` で囲む既存の操作を維持すること。
3. 本文に全角カッコ、入れ子のカッコ、閉じ忘れなどの問題がある場合、システムは投稿ボタンに到達する前に原因と修正方法が分かる警告を表示すること。
4. シンプルモードを利用する場合、システムは通常モードとの違いと切り替え時の入力破棄を投稿者が理解できるようにすること。
5. 連続した伏せ字記号を省略する設定を利用する場合、システムは Bluesky 上の表示にのみ影響することを投稿者が理解できるようにすること。

### 要件 4: 投稿前チェックを Skyblur 固有の公開事故防止体験にする

**ユーザーストーリー:** 投稿者として、投稿前に Bluesky 上で公開される内容と Skyblur 上で閲覧可能になる内容を比較したい。そうすることで、伏せ字と補足情報の公開先を誤解せずに投稿できる。

#### 受け入れ基準

1. 本文が入力されたとき、システムは Bluesky に投稿される伏せ字後の本文をプレビューとして表示すること。
2. 補足が入力されたとき、システムは補足が Bluesky には反映されず Skyblur 上でのみ表示されることを明示すること。
3. 投稿者が公開範囲を変更したとき、システムは伏せた本文と補足の閲覧条件がどのように変わるかを投稿前に確認できるようにすること。
4. 投稿前の確認領域では、システムは少なくとも Bluesky 側の表示、Skyblur 側の内容、選択中の公開範囲を同時に把握できるようにすること。
5. プレビューは既存の投稿データ生成ルールと整合し、実際の Bluesky 投稿本文と異なる期待を与えないこと。
6. 投稿前チェックでは、システムは読める人だけでなく、読めない人にどのように見えるかも確認できるようにすること。
7. MVP の編集時に同じ保存形式内で設定を変更する場合、システムは変更後に誰が読めるかを表示すること。
8. 投稿前チェックの「誰が読めるか」は、表示用の文言だけでなく、実際の保存形式とサーバー側認可条件と矛盾しないロジックに基づくこと。
9. MVP では保存形式をまたぐ公開範囲変更を行わないため、編集時にその制約がある場合は投稿前チェックまたは該当ステップで分かるようにすること。

### 要件 5: 公開範囲の選択を理解しやすく整理する

**ユーザーストーリー:** 投稿者として、伏せた内容を誰に見せるかを意味ベースで選びたい。そうすることで、公開範囲を誤って選ぶリスクを下げられる。

#### 受け入れ基準

1. 公開範囲を選択するとき、システムは `全体`、`ログイン`、`パスワード`、`フォロワー限定`、`フォロー中限定`、`相互フォロー限定`、`リスト限定` の既存選択肢を維持すること。
2. 投稿者が公開範囲を選択したとき、システムは選択中の範囲について「誰が閲覧できるか」「ログインが必要か」「Bluesky 上の非伏せ字部分はどう扱われるか」を説明すること。
3. 投稿者が `パスワード` を選択した場合、システムはパスワード入力欄と空白文字の禁止を分かりやすく表示すること。
4. 投稿者が `リスト限定` を選択した場合、システムは閲覧を許可するリストの選択が必須であることを表示し、未選択のまま投稿できないようにすること。
5. 投稿者が関係性ベースの公開範囲を選択した場合、システムは閲覧には Skyblur へのログインが必要であることを明示すること。
6. 投稿者が公開範囲を選択するとき、システムは機能名だけでなく「誰が伏せた部分と補足を読めるか」が分かる説明を表示すること。

### 要件 6: 編集は同じ保存形式内に絞り、パスワード投稿は編集前に解除する

**ユーザーストーリー:** 既存投稿を編集する投稿者として、投稿の保存方式を大きく変えずに本文や補足を直したい。パスワード投稿の場合は、編集前に解除用パスワードで安全に復号したい。そうすることで、投稿画面刷新の MVP でも既存投稿を安心して編集できる。

#### 受け入れ基準

1. MVP の既存投稿編集では、公開範囲の保存形式をまたぐ変更を行わないこと。
2. 非パスワード投稿を編集する場合、システムは現在の保存形式を保ったまま本文、補足、同じ保存形式内の基本設定を編集できること。
3. 現在の公開範囲が `パスワード` の既存投稿を編集する場合、システムは編集画面を表示する前に解除用パスワードの確認と復号を完了し、復号に失敗した状態では編集画面に進ませないこと。
4. パスワード投稿を編集する場合、システムは復号済み本文と補足を編集対象として扱い、保存時は `パスワード` 形式のまま暗号化、Blob upload、Skyblur record 更新の既存フローを維持すること。
5. パスワード投稿の編集で、解除用パスワードまたは必要な復号材料がない場合、システムは暗号化本文・補足を平文として再保存しないこと。
6. MVP では、非パスワード投稿から `パスワード` への変更、および `パスワード` 投稿から非パスワード公開範囲への変更を保存しないこと。
7. 保存形式をまたぐ公開範囲変更が必要な場合、システムは MVP では未対応であることを表示し、必要に応じて別の公開範囲で新規投稿として作り直す導線を提供すること。
8. パスワード、復号済み本文、復号済み補足、暗号化前本文は、権限のない表示領域、ログ、エラー、URL、サーバー側の不要な途中状態に残らないようにすること。
9. パスワード投稿の作成・編集途中に必要な本文、補足、パスワード、暗号化材料は、投稿者本人のブラウザ内の一時保持に限って許容すること。
10. パスワード投稿の一時保持データは、投稿成功時、投稿者が明示的に破棄した時、ログアウトまたはセッション変更時に必ずクリアすること。
11. MVP では record の revision、updatedAt、cid 相当の競合検知は行わない。ただし、編集保存では対象 record の有無と dirty 判定に基づき、未変更の threadgate/postgate write を発行せず、gate 保存失敗が本文・補足保存を巻き込まないこと。
12. 投稿前チェックでは、システムは MVP で保存される変更にもとづいて「誰が読めるか」を表示すること。

### 要件 7: 返信・引用制限を主要投稿フローから分離して扱う

**ユーザーストーリー:** 投稿者として、必要な場合だけ返信や引用の制限を設定したい。そうすることで、通常の投稿では画面を複雑にせず、必要な投稿では Bluesky 上の会話制御を利用できる。

#### 受け入れ基準

1. 投稿者が通常の投稿作成を行うとき、システムは返信先選択と返信・引用制限が本文作成や公開範囲選択を妨げないように表示すること。
2. 投稿者が返信先を設定する場合、システムは既存の返信先検索・選択機能を維持すること。
3. 投稿者が返信制限を設定する場合、システムはメンションしたユーザー、フォロー中のユーザー、フォロワー、引用許可の既存選択肢を維持すること。
4. 返信・引用制限は Bluesky 上でのみ有効であるため、システムは Skyblur の閲覧制御とは別の設定であることを分かるようにすること。
5. 投稿者が設定を変更しない場合、システムは既存と同じ既定値で投稿できること。

### 要件 8: モバイルで投稿完了までの操作距離を短くする

**ユーザーストーリー:** モバイルで投稿するユーザーとして、長い投稿画面でも現在の状態を確認しながらすぐ投稿または戻る操作をしたい。そうすることで、設定の確認漏れや操作迷いを減らせる。

#### 受け入れ基準

1. モバイル幅で投稿画面を表示したとき、システムは本文入力、プレビュー、公開範囲、投稿アクションが画面幅に収まり、テキストやボタンが重ならないようにすること。
2. モバイル幅で画面が縦に長くなる場合、システムは投稿アクションと戻る操作を見失いにくい位置または構造で提供すること。
3. モバイル幅で公開範囲や返信制限の選択肢が折り返される場合、システムは各選択肢のラベル、アイコン、選択状態が判別できるようにすること。
4. 投稿者が入力や設定を変更したとき、システムはモバイルでも投稿可能・不可の理由を確認できるようにすること。
5. モバイル表示は主要対象ブラウザである Safari/WebKit で使用できること。

### 要件 9: 既存の投稿作成・編集機能との互換性を保つ

**ユーザーストーリー:** 既存ユーザーとして、投稿画面が刷新されてもこれまで使っていた投稿作成・編集・公開範囲設定を失わずに使いたい。そうすることで、UI 変更後も安心して Skyblur を継続利用できる。

#### 受け入れ基準

1. 新規投稿を作成する場合、システムは既存と同じ投稿本文、伏せ字本文、補足、公開範囲、返信、返信制限を保存できること。
2. 既存投稿を編集する場合、システムは既存の編集フローを維持し、Skyblur 側の本文・補足・基本設定の更新が可能であること。
3. 既存投稿を編集する場合、システムは返信・引用制限を更新できること。ただし、認可 scope や保存対象 record の状態により更新できない場合は、本文・補足の保存と混同しないエラーまたは案内を表示すること。
4. 既存投稿を編集する場合、MVP では既存 Bluesky post 本体の本文更新は対象外とする。編集時の投稿前チェックでは Bluesky 側本文を表示せず、Bluesky 側本文や OGP/card が更新対象外であることを投稿者が理解できるようにすること。
5. パスワード保護投稿を作成または編集する場合、システムは暗号化、Blob upload、Skyblur record 作成・更新のフローを維持すること。
6. 制限付き投稿を作成する場合、システムは制限付き本文・補足の保存と Skyblur record 作成の既存フローを維持すること。
7. 投稿途中の入力内容は、ステップ移動や投稿失敗によって失われないこと。
8. 未保存の入力や設定変更がある状態でブラウザバック、リロード、別ページ遷移、タブクローズを行う場合、システムは離脱確認を表示すること。
9. 本刷新では、永続的な下書き一覧、下書き同期、下書き共有は対象外とすること。
10. 既存の一時保存または復元の仕組みがある場合、システムはその復元・破棄フローに回帰を起こさないこと。
11. 投稿処理中に暗号化、Blob upload、制限付き保存、applyWrites のいずれかが失敗した場合、システムは秘密情報を漏らさず、投稿者が失敗を理解できるエラーを表示すること。

### 要件 10: 実画面の E2E で回帰を検知する

**ユーザーストーリー:** 開発者として、投稿画面刷新後も実際の production route 上の投稿フローが動くことを確認したい。そうすることで、見た目だけでなく実導線の動作回帰を検知できる。

#### 受け入れ基準

1. 投稿画面の E2E は `/console` を起点に、専用の投稿作成・編集 route へ遷移するログイン済み production flow を対象にすること。
2. 既存の `/console` inline 投稿画面を前提にした E2E は、新しい専用 route を対象にする production flow へ移行すること。
3. E2E は既存投稿画面 E2E の移行を対象に含め、3 ステップの遷移、本文入力、伏せ字指定、Bluesky プレビュー、Skyblur Check、補足、公開範囲、投稿ボタンの有効・無効を確認すること。
4. E2E は新規投稿の既存 visibility 互換として、主要な visibility の保存、password 新規投稿、リスト未選択エラーを確認すること。
5. E2E は編集 route で同じ保存形式内の編集、返信・引用制限の更新可否、password 投稿の編集前 unlock gate、編集時に更新対象・非更新対象が投稿者に表示されることを確認すること。編集時の投稿前チェックでは Bluesky 側本文が表示されないことも確認すること。
6. E2E は desktop と mobile の主要表示で、レイアウト破綻や操作不能がないことを確認すること。
7. E2E は Safari/WebKit を重要対象として維持すること。
8. E2E は production route の実ユーザーフローを対象にし、人工的な preview/test-only 画面だけを根拠に投稿画面刷新の品質判断をしないこと。
9. visibility、編集条件、端末、ブラウザ、エラー条件の全組み合わせ網羅は E2E のみで担わず、unit test、integration test、E2E で分担すること。
10. 保存形式をまたぐ password 区分変更や公開範囲移行の E2E matrix は MVP の対象外とし、次フェーズの要件として分離すること。

## 非機能要件

### コード構造とモジュール性

- **単一責任の原則**: 投稿画面のステップ制御、本文入力、公開範囲説明、パスワード投稿の解除・編集、投稿前サマリー、返信・引用設定、バリデーション表示は、責務が分かる単位に整理すること。
- **モジュール設計**: 既存の `CreatePost.tsx` に機能を集約しすぎず、再利用できる表示部品や選択肢定義は既存パターンに沿って分離すること。
- **ロケール管理**: 投稿画面刷新で追加する文言は、既存の `ja.ts` / `en.ts` へ無秩序に追加せず、MVP では投稿 composer 専用 prefix で管理し、ファイル分割は次フェーズで扱える状態にすること。
- **ロケール互換性**: 既存の `locale.CreatePost_*` 参照を壊さない移行手順を取り、必要に応じて段階的に prefix 整理または名前空間化すること。
- **ロケール型安全性**: 日本語と英語でメッセージキーの欠落や不一致が起きないよう、MVP では追加キーの対応関係を明示し、共通の型または検証の強化は次フェーズで扱える状態にすること。
- **文言責務の分離**: ステップ名、Skyblur Check、公開範囲説明、パスワード投稿の解除・編集、エラー、離脱確認などの文言は、UI 実装から追いやすい単位で定義すること。
- **依存関係管理**: 新しい依存関係は原則追加せず、既存の Mantine、lucide-react、zustand、atcute の範囲で実現すること。
- **明確なインターフェース**: 投稿作成に必要な既存 state、locale、XRPC agent、visibility 値の契約を変える場合は、影響範囲を明確にすること。

### パフォーマンス

- 投稿画面の通常入力、公開範囲切り替え、プレビュー更新は、体感上遅延なく反映されること。
- 外部 API 呼び出しが必要な返信先取得やリスト取得は、投稿本文の入力体験を阻害しないこと。
- 投稿処理中は既存と同等以上に進行状況を段階的に表示し、二重投稿を防ぐこと。

### セキュリティ

- 伏せた本文、補足、パスワード、暗号化前の本文は、権限のない表示領域、エラーメッセージ、ログに漏らさないこと。
- 公開範囲の UI 表示は、実際のサーバー側認可と矛盾しないこと。
- パスワード保護、制限付き投稿、リスト限定の保存・取得・編集・公開範囲変更に関する deny-by-default 方針を維持すること。

### 信頼性

- 既存の OAuth session、XRPC proxy、applyWrites、制限付き投稿保存、暗号化、Blob upload、公開範囲変更時の保存整合性に回帰を起こさないこと。
- 投稿作成中に失敗した場合、投稿者が再試行または修正できる状態を維持すること。
- 投稿画面刷新後も一時保存、復元、編集、戻る確認の既存体験を維持すること。

### 使いやすさ

- 投稿者が「何が Bluesky に公開されるか」「何が Skyblur だけで見えるか」「誰が読めるか」を投稿前に理解できること。
- 主要操作は mobile と desktop の両方で見つけやすく、テキスト、ボタン、選択肢が重ならないこと。
- 画面内説明は必要十分にし、フォームの各段階で投稿者の判断に直接役立つ文言にすること。
- 既存ユーザーが大きく迷わないよう、機能名や公開範囲の意味は既存の表現と整合させること。
