# 要件定義書

## 概要

Skyblur は現在、伏せ字本文と補足情報の公開範囲として、全体公開、ログイン必須、パスワード必須、フォロワー限定、フォロー中限定、相互フォロー限定をサポートしている。この機能では、新たに「リスト限定」の公開範囲を追加し、投稿者が自分の Bluesky リストを選択して、そのリストのメンバーだけが Skyblur 上の伏せ字本文と補足情報を閲覧できるようにする。

リスト限定の閲覧可否判定には `GET /xrpc/blue.microcosm.links.getBacklinks` を使用する。UI では投稿者自身が所有するリストだけを選択可能にし、既存の制限付き公開範囲と同じ作成・編集・閲覧体験を維持する。Skyblur の Lexicon も更新し、レコードと XRPC がリスト限定公開に必要な状態を表現・検証できるようにする。

## プロダクト方針との整合性

Skyblur は、Bluesky 上では伏せ字として共有しつつ、Skyblur 上でだけ条件付きで原文や補足情報を閲覧できるようにするアプリケーションである。リスト限定公開は、投稿者が閲覧者をより細かく制御できるようにし、ネタバレ・センシティブ情報・限定共有の安全性と使いやすさを高める。

## 要件

### 要件 1

**ユーザーストーリー:** Skyblur の投稿者として、自分の Bluesky リストを投稿の公開範囲として選択したい。そうすることで、そのリストのメンバーだけに隠された Skyblur コンテンツを見せられる。

#### 受け入れ基準

1. 投稿者が公開設定 UI を開いたとき、システムは既存の公開範囲に加えてリスト限定の選択肢を表示すること。
2. 投稿者がリスト限定を選択したとき、システムは許可対象のリストを選ぶためのリスト選択 UI を表示すること。
3. リスト選択 UI を表示するとき、システムは単純なテキストだけのドロップダウンではなく、リスト名、説明または用途が分かる補助情報、メンバー数など取得可能な要約情報、選択状態が視覚的に分かる表示を含めること。
4. リスト選択 UI を表示するとき、システムは Skyblur の既存 UI と調和しつつ、公開範囲を決める重要な操作として目立ち、投稿者が「このリストを選ぶ」と確信できる polished な見た目にすること。
5. リスト選択 UI を表示するとき、システムは読み込み中、取得失敗、リストなし、選択済み、未選択エラーの各状態を、レイアウトが崩れない形で明確に表示すること。
6. リスト選択 UI はモバイルとデスクトップの両方で操作しやすく、長いリスト名や説明があってもテキストがボタン・カード・選択肢からはみ出さないこと。
7. システムが選択可能なリストを表示するとき、ログイン中の投稿者 DID が所有するリストだけを含めること。
8. ログイン中の投稿者に選択可能なリストが存在しない場合、システムは所有リストがないことを説明し、リスト限定での投稿を禁止すること。
9. 投稿者が自分のリストを選択したとき、システムは選択されたリスト識別子を Skyblur 投稿データに保存すること。
10. 投稿者がリストを選択せずにリスト限定コンテンツを投稿しようとしたとき、システムは投稿をブロックし、検証エラーを表示すること。

### 要件 2

**ユーザーストーリー:** Skyblur の閲覧者として、自分が選択されたリストのメンバーである場合だけリスト限定投稿を閲覧したい。そうすることで、投稿者が意図した相手だけが非公開コンテンツにアクセスできる。

#### 受け入れ基準

1. 未ログインの閲覧者がリスト限定投稿を要求したとき、システムは既存の制限付き公開範囲と同様に、伏せ字の公開本文、空の補足情報、ログイン必須の応答を返すこと。
2. 投稿者本人が自分のリスト限定投稿を要求したとき、システムはリストメンバー判定を行わずに閲覧を許可すること。
3. ログイン済みの投稿者以外の閲覧者がリスト限定投稿を要求したとき、システムは `GET /xrpc/blue.microcosm.links.getBacklinks` を呼び出して、閲覧者を含む `app.bsky.graph.listitem` レコード候補を取得すること。
4. `blue.microcosm.links.getBacklinks` を呼び出すとき、システムは閲覧者 DID を URL エンコードした `subject` クエリパラメータとして含めること。
5. `blue.microcosm.links.getBacklinks` を呼び出すとき、システムはリストメンバーシップの対象 DID を表す `source=app.bsky.graph.listitem:subject` を含めること。
6. 特定投稿者のリストだけを判定するとき、システムは投稿者 DID を `did` クエリパラメータとして含めること。
7. `blue.microcosm.links.getBacklinks` の戻り値を処理するとき、システムは `total` と `records` を検証し、`records` 内の `{ did, collection, rkey }` から候補の `app.bsky.graph.listitem` レコードを取得して、その `list` が選択リスト URI と一致することを確認すること。
8. 候補レコードの中に `subject` が閲覧者 DID、かつ `list` が選択リスト URI と一致する list item が存在する場合、システムは閲覧者を許可すること。
9. 候補レコードが存在しない場合、または候補レコードの `list` が選択リスト URI と一致しない場合、システムはリスト限定用の権限不足エラーで閲覧を拒否すること。
10. backlink 取得、候補 list item 取得、または応答検証が失敗した場合、システムは制限付きコンテンツを表示せず、復旧可能な認可エラーを返すこと。
11. エンドポイントリクエストを組み立てるとき、システムは `limit` の最大値 100 と `reverse=false` を使用し、`total` が `records.length` を超えるなど候補を網羅できない場合は、許可せず復旧可能な認可エラーとして扱うこと。

### 要件 3

**ユーザーストーリー:** API クライアントとして、Skyblur Lexicon がリスト限定公開とリストメタデータを表現してほしい。そうすることで、新しいアクセス制御状態をレコードと XRPC 呼び出しで検証できる。

#### 受け入れ基準

1. `uk.skyblur.post` レコードスキーマを更新するとき、リスト限定を表す公開範囲値を許可すること。
2. 投稿がリスト限定である場合、レコードスキーマは選択されたリスト URI を保持するフィールドを提供すること。
3. `uk.skyblur.post.store` 入力スキーマを更新するとき、リスト限定コンテンツの保存を許可し、リスト限定の場合は選択されたリスト URI を必須にすること。
4. `uk.skyblur.post.getPost` がリスト限定の認可エラーを返すとき、出力スキーマはリスト限定の公開範囲値とリスト用エラーコードを表現できること。
5. Lexicon JSON ファイルを変更したとき、frontend と backend の生成済み Lexicon TypeScript 型を再生成する、または JSON スキーマと整合するように更新すること。
6. リスト限定公開で有効なリスト URI が指定されていない場合、Lexicon またはアプリケーション検証により、制限付きコンテンツが保存される前にリクエストを拒否すること。

### 要件 4

**ユーザーストーリー:** Skyblur の投稿者および閲覧者として、リスト限定公開が UI 上で明確に表示されてほしい。そうすることで、隠されたコンテンツに誰がアクセスできるのか理解できる。

#### 受け入れ基準

1. 投稿がリスト限定であるとき、投稿作成、編集、投稿詳細、投稿一覧 UI はリスト限定用の公開範囲ラベルを表示すること。
2. 選択されたリストの表示名が取得できるとき、UI はリスト限定コントロールまたは投稿の公開範囲表示の近くにリスト名を表示すること。
3. 選択されたリスト名を読み込めないとき、UI は安定したリスト識別子にフォールバックし、操作フローを壊さないこと。
4. 閲覧者がリスト限定投稿の閲覧権限を持たないとき、UI は汎用システムエラーではなく、リスト限定用の権限不足メッセージを表示すること。
5. 閲覧者が未ログインで投稿がリスト限定であるとき、UI は他の制限付き公開範囲と一貫したログイン必須メッセージを表示すること。
6. UI 文言を更新するとき、日本語と英語の両方のロケールに、リスト限定ラベル、説明、検証メッセージ、認可メッセージを追加すること。

### 要件 5

**ユーザーストーリー:** メンテナーとして、リスト限定公開が既存の制限付き投稿の挙動を保ってほしい。そうすることで、新機能追加によって既存の公開範囲が壊れないようにできる。

#### 受け入れ基準

1. 投稿が全体公開、ログイン必須、パスワード必須、フォロワー限定、フォロー中限定、相互フォロー限定のいずれかを使用しているとき、既存の作成、保存、取得、表示の挙動は変わらないこと。
2. リスト限定コンテンツを保存するとき、システムはフォロワー限定、フォロー中限定、相互フォロー限定と同じ制限付きコンテンツ保存の保護を使用すること。
3. 編集時に投稿がリスト限定から非制限公開範囲へ変更されたとき、システムは既存の制限付き公開範囲のクリーンアップと一貫した形で保存済み制限コンテンツを削除すること。
4. 編集時に投稿が非リスト公開範囲からリスト限定へ変更されたとき、システムは制限付きコンテンツと選択リストメタデータを一貫した形で保存すること。
5. テストを追加するとき、投稿者本人の閲覧成功、リストメンバーの閲覧成功、非メンバーの拒否、未ログインの拒否、リスト未選択の投稿ブロック、既存公開範囲の回帰をカバーすること。

## 非機能要件

### コード構造とモジュール性
- **単一責任の原則**: リストメンバー判定、リスト所有者フィルタリング、Lexicon スキーマ更新、UI コントロールは、それぞれ明確な責務を持つモジュールまたは関数に分離すること。
- **モジュール設計**: リスト限定の実装は、可能な限り既存の制限付き公開範囲の処理を再利用し、並行した別フローを作らないこと。
- **依存関係管理**: 新しい依存関係は、既にプロジェクトで使われているもの、または設計上明確に必要なものに限定すること。
- **明確なインターフェース**: backend の認可処理は、閲覧者がリスト限定投稿にアクセスできるかを判定する明確な契約を持つこと。

### パフォーマンス
- 認可判定では、メンバーシップ判断に必要な最小限の backlink データだけを要求すること。
- UI はリスト名の読み込み中に投稿エディタ全体をブロックしないこと。リスト選択部分には読み込み状態を表示してよい。
- 同一ページセッション内で繰り返されるリストメタデータ取得は、実用的な範囲でキャッシュまたは再利用すること。

### セキュリティ
- サーバーは、認証、リストメタデータ取得、backlink 判定のいずれかに失敗した場合、リスト限定公開を deny-by-default として扱うこと。
- サーバーはクライアントから渡されたリスト所有情報を信頼しないこと。選択可能または保存済みのリスト制限が投稿者所有リストを参照していることを検証すること。
- サーバーは `blue.microcosm.links.getBacklinks` のリクエストを安全に構築し、`subject` を必ず URL エンコードすること。
- 実装は既存の SSRF 対策と安全な XRPC ルーティングを維持すること。
- エラー応答は、隠された本文、補足情報、パスワード、非公開のリストメンバーシップ詳細を漏らさないこと。

### 信頼性
- backlink エンドポイントが利用できない場合、または不正な応答を返した場合、リスト限定コンテンツは非表示のままにし、認可を確認できなかったことを応答で伝えること。
- この機能は、直接の投稿ページ、投稿一覧、API/XRPC アクセスで一貫して動作すること。
- 生成済み Lexicon 型は frontend と backend の両方で同期されていること。

### 使いやすさ
- リスト選択 UI は、投稿者自身のリストだけが選択可能であることを明確に伝えること。
- リスト選択 UI は、公開範囲選択の一部として自然に見えつつ、情報密度、余白、アイコン、選択状態、エラー状態が丁寧に設計された見た目にすること。
- リスト選択 UI は、リスト数が少ない場合も多い場合も選びやすく、検索または絞り込みが必要な件数になった場合の操作性を考慮すること。
- 検証エラーは、リストを選択するなど投稿者が次に取るべき行動を説明すること。
- 閲覧者向けの拒否メッセージは、実装詳細を明かさずに理解できる内容にすること。
